  <name>Task 1: Create lib/markdown-split.js with all 6 exported functions</name>
  <files>lib/markdown-split.js</files>
  <action>
    Create `lib/markdown-split.js` following established codebase conventions: `'use strict'` at top, JSDoc module header, require Node built-ins first then local modules, section separators (`// ───`), `module.exports` at bottom.

    **Imports:**
    - `fs`, `path` from Node built-ins
    - `safeReadFile` from `./utils`

    **Constants:**
    - `INDEX_MARKER = '<!-- GRD-INDEX -->'` — magic comment identifying an index file
    - `DEFAULT_TOKEN_THRESHOLD = 25000` — token count above which splitting is triggered
    - `PARTIAL_SUFFIX_PATTERN = /-part(\d+)\.md$/` — regex matching partial filenames
    - `CHARS_PER_TOKEN = 4` — heuristic: ~4 characters per token (GPT tokenizer average)

    **Functions to implement (6 exported):**

    1. `estimateTokens(content)` — Returns estimated token count for a string.
       - Implementation: `Math.ceil(content.length / CHARS_PER_TOKEN)`
       - This is a deliberate simplification. Real tokenizers vary by model, but character-based heuristic is sufficient for threshold detection. The 4 chars/token ratio is conservative (errs toward splitting earlier rather than later).
       - Returns: `number`

    2. `findSplitBoundaries(content, targetPartCount)` — Finds heading-level line offsets where the content should be split.
       - Scan content for lines matching `^#{1,2}\s` (h1 and h2 headings only — smaller headings don't represent logical boundaries)
       - Compute target size per part = totalLength / targetPartCount
       - Greedily assign headings to parts: accumulate content until crossing the target boundary, then split at the nearest heading
       - Return an array of character offsets where splits should occur
       - If no headings found, fall back to splitting at blank line boundaries (`\n\n`)
       - Edge case: if content has fewer headings than targetPartCount, reduce the number of splits to match available boundaries
       - Returns: `number[]` (array of split offsets)

    3. `splitMarkdown(content, options = {})` — Core splitting function. Pure function (no I/O).
       - `options.threshold` — token threshold (default: DEFAULT_TOKEN_THRESHOLD)
       - `options.basename` — base name for partial files (default: 'DOC')
       - **Guard 1:** If `isIndexFile(content)` returns true, return `{ split_performed: false, reason: 'already_split' }` — idempotency
       - **Guard 2:** If `estimateTokens(content)` <= threshold, return `{ split_performed: false, reason: 'below_threshold' }`
       - Compute target part count: `Math.ceil(estimateTokens(content) / (threshold * 0.8))` — use 80% of threshold per part to give headroom
       - Call `findSplitBoundaries(content, targetPartCount)` to get split offsets
       - Slice content at boundaries into parts array
       - Generate index content:
         ```
         <!-- GRD-INDEX -->
         # {basename} (Split Index)

         This file has been automatically split into partials by GRD.

         ## Partials

         - [{basename}-part1.md](./{basename}-part1.md)
         - [{basename}-part2.md](./{basename}-part2.md)
         ...
         ```
       - Return: `{ split_performed: true, index_content: string, parts: Array<{ filename: string, content: string }> }`
       - Each part: `{ filename: '{basename}-part{N}.md', content: string }` (N is 1-based)

    4. `isIndexFile(content)` — Returns true if the content contains the GRD index marker.
       - Implementation: `typeof content === 'string' && content.includes(INDEX_MARKER)`
       - Returns: `boolean`

    5. `reassembleFromIndex(indexPath)` — Reads an index file and all its partials from disk, returns concatenated content.
       - Read the index file content using `safeReadFile(indexPath)`
       - If content is null or `!isIndexFile(content)`, return null (not an index file)
       - Parse partial links from the index: match all `[...](./BASENAME-partN.md)` patterns
       - For each partial link, resolve the path relative to the index file's directory
       - Read each partial using `safeReadFile`
       - If any partial is missing, throw an error with the missing filename
       - Concatenate all partial contents in order (join with `\n\n`)
       - Returns: `string` (the reassembled content)

    6. `readMarkdownWithPartials(filePath)` — Transparent reader: if the file is a GRD index, reassemble; otherwise return content as-is.
       - Read file content using `safeReadFile(filePath)`
       - If content is null, return null
       - If `isIndexFile(content)`, call `reassembleFromIndex(filePath)` and return result
       - Otherwise, return content as-is
       - This is the function that Plan 54-02 will wire into existing readers
       - Returns: `string | null`

    **Module exports:** All 6 functions. Also export `INDEX_MARKER` and `DEFAULT_TOKEN_THRESHOLD` as named constants for test access.

    **Important design notes:**
    - `splitMarkdown` is a pure function — it computes the split but does NOT write files. The caller (CLI command in Plan 54-02) handles disk I/O. This keeps the core logic testable without filesystem mocks.
    - `reassembleFromIndex` and `readMarkdownWithPartials` DO read from disk because they need to resolve partial file paths.
    - Heading detection regex: `^#{1,2}\s` — only split at h1/h2 boundaries for coherent chunks. Splitting at h3+ would create too many tiny partials.
    - The 80% headroom factor in part count calculation prevents boundary conditions where the last partial is disproportionately large.
