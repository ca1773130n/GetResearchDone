  <action>
    **Part A: Harden ghExec (GitHub tracker)**

    The `createGitHubTracker` function (starting around line 4572) contains a `ghExec` helper
    that currently takes a single command string and passes it to the unsafe shell execution.
    All calls in this tracker construct shell command strings with user-provided titles, bodies,
    and labels via string interpolation. This is a command injection vector.

    1. Rewrite `ghExec` to take an array of arguments and use execFileSync:
       ```
       function ghExec(args) {
         try {
           return execFileSync('gh', args, {
             cwd, encoding: 'utf-8', timeout: 30000, stdio: 'pipe'
           }).trim();
         } catch { return null; }
       }
       ```

    2. Update `ghAvailable`: call ghExec(['--version'])

    3. Update `createPhaseIssue`: Build argument array:
       - Start with ['issue', 'create', '--title', title, '--body', body || '', '--label', 'epic']
       - Append '--label', labelName for each label
       - Append '--assignee', assignee if present
       - Call ghExec(args)
       - Remove all manual escaping (safeTitle, safeBody string replacements)

    4. Update `createTaskIssue`:
       - ghExec(['issue', 'create', '--title', title, '--body', bodyText, '--label', 'task'])
       - ghExec(['sub-issue', 'add', parentNum, '--child', issueRef])

    5. Update `updateIssueStatus`:
       - ghExec(['issue', 'edit', num, '--remove-label', sl])
       - ghExec(['issue', 'edit', num, '--add-label', label])
       - ghExec(['issue', 'close', num])

    6. Update `addComment`:
       - ghExec(['issue', 'comment', num, '--body', markdownBody])
       - Remove safeBody escaping

    7. Update the `gh auth status` check (in tracker get-config, around line 4716):
       - Replace with execFileSync('gh', ['auth', 'status'], { cwd, encoding: 'utf-8', timeout: 10000, stdio: 'pipe' })

    **Part B: Add git operation whitelist**

    8. Define a git subcommand whitelist constant near the top of the file:
       ```
       const GIT_ALLOWED_COMMANDS = new Set([
         'add', 'commit', 'log', 'status', 'diff', 'show',
         'rev-parse', 'cat-file', 'check-ignore', 'ls-files',
         'branch', 'checkout', 'merge', 'rebase', 'cherry-pick',
         'tag', 'stash', 'remote', 'fetch', 'pull',
       ]);
       const GIT_BLOCKED_COMMANDS = new Set(['config', 'push', 'clean']);
       const GIT_BLOCKED_FLAGS = new Set(['--force', '-f', '--hard', '--delete', '-D']);
       ```

    9. Add whitelist enforcement at the top of execGit. Extend the function signature to accept
       an optional third parameter `opts = {}`:
       - Extract the subcommand (args[0])
       - If subcommand is in GIT_BLOCKED_COMMANDS and opts.allowBlocked is not true, return an
         error result with stderr explaining the block
       - Scan all args: if any arg is in GIT_BLOCKED_FLAGS and opts.allowBlocked is not true,
         return an error result
       - This is the secure default. No existing call site passes blocked commands or flags,
         so zero breaking changes.

    3. **Git Operation Whitelist** - List the allowed commands (add, commit, log, status, diff,
       show, rev-parse, cat-file, check-ignore, ls-files, branch, checkout, merge, rebase,
       cherry-pick, tag, stash, remote, fetch, pull). List the blocked commands (config, push,
       clean) and blocked flags (--force, -f, --hard, --delete, -D). Explain override mechanism.

    4. **Input Validation** - Document the three validators:
       - Phase names: regex pattern, path traversal rejection
       - File paths: project directory containment, null byte rejection
       - Git refs: safe character set, leading dash rejection

    5. **Environment Safety** - process.env.HOME fallback, .planning/ scope

    6. **GitHub CLI Security** - execFileSync usage, argument-based API,
       non-blocking failure mode

    7. **Reporting Vulnerabilities** - Instructions for reporting security issues

    Keep the document concise and factual. No marketing language. Target 40-60 lines.
