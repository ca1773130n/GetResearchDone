  <name>Task 1: Add isolation_mode and main_repo_path to cmdInitExecutePhase</name>
  <files>lib/context.js, tests/unit/context.test.js</files>
  <action>
    In lib/context.js, within cmdInitExecutePhase, add two new fields to the result object:

    1. **isolation_mode** — Derive as follows:
       - If `branching_strategy` is `'none'`: set to `'none'` (no isolation at all)
       - Else if `getBackendCapabilities(backend).native_worktree_isolation` is true: set to `'native'`
       - Else: set to `'manual'`

    2. **main_repo_path** — Set to `fs.realpathSync(cwd)` when branching_strategy is not 'none', or `null` when branching_strategy is 'none'.

    Place these fields near the existing `native_worktree_available` field for logical grouping.

    The config's `branching_strategy` value comes from `config.branching_strategy` which is already loaded (it's used for `branching_strategy` field in the result). The backend capabilities are already available via `getBackendCapabilities(backend)`.

    In tests/unit/context.test.js, add tests:
    - Verify isolation_mode is 'native' when backend is 'claude' and branching_strategy is 'phase'
    - Verify isolation_mode is 'manual' when backend is 'codex' and branching_strategy is 'phase'
    - Verify isolation_mode is 'none' when branching_strategy is 'none' regardless of backend
    - Verify main_repo_path is a non-null string when branching_strategy is 'phase'
    - Verify main_repo_path is null when branching_strategy is 'none'

    Important: `fs` is already required at the top of context.js (it's used elsewhere). Use `fs.realpathSync(cwd)` to resolve macOS symlinks consistently with worktree.js's worktreePath function.
  </action>
  <verify>
    npx jest tests/unit/context.test.js --no-coverage (Level 2: Proxy — tests validate field values under different backend/config combos)
  </verify>
  <done>

  </done>

       - If `native_worktree_available` is false: preserve existing behavior (compute worktree_path as before)

    3. The function signature may need an additional parameter or it may derive native_worktree_available from the capabilities. Check the existing function signature and adapt accordingly. The most natural approach: pass a `nativeWorktreeAvailable` boolean parameter (defaulting to false for backward compatibility) or derive it from the backend capabilities that are already available in the context.

    In tests/unit/parallel.test.js, add tests:
    - When nativeWorktreeAvailable=true, phase contexts have worktree_path=null and native_isolation=true
    - When nativeWorktreeAvailable=false, phase contexts have worktree_path set to a valid path (existing behavior preserved)
    - Existing parallel tests continue to pass without modification
