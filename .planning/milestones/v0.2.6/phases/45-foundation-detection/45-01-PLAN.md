  <name>Task 2: Expose native_worktree_available in cmdInitExecutePhase and write tests</name>
  <files>lib/context.js, tests/unit/context.test.js</files>
  <action>
    In `lib/context.js`, in the `cmdInitExecutePhase` function, add a `native_worktree_available` field to the result object. Place it in the "Config flags" section, near the `branching_strategy` field.

    The value should be computed as:
    ```javascript
    native_worktree_available: getBackendCapabilities(backend).native_worktree_isolation === true
    ```

    This is a simple boolean derived from the backend capabilities. It is `true` only when the backend is Claude Code. The branching_strategy check is NOT needed here — that decision belongs to the orchestrator (Phase 46). This field reports capability, not policy.

    Also add `native_worktree_available` to the `cmdInitExecuteParallel` function if it exists (check `lib/context.js` for this function), using the same logic.

    Then write tests in `tests/unit/context.test.js`:
    - Test that `cmdInitExecutePhase` result includes `native_worktree_available` property
    - Test that when backend is `claude`, `native_worktree_available` is `true`
    - Test that when backend is not `claude` (e.g., config override to `codex`), `native_worktree_available` is `false`

    These tests should follow the existing pattern in `context.test.js` for testing init fields (look at how `branching_strategy`, `webmcp_available` etc. are tested).
