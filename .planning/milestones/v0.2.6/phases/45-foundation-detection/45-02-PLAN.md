Register WorktreeCreate and WorktreeRemove hooks in the Claude Code plugin manifest and implement handler functions.

Purpose: When Claude Code creates or removes a worktree natively (via `isolation: worktree`), GRD's hooks fire to optionally rename the branch to GRD's naming convention and log the lifecycle event. This enables GRD to track and manage worktrees it did not create itself.

Output: Updated plugin.json with hook registrations, handler functions in lib/worktree.js, routing in bin/grd-tools.js, and tests.

  <name>Task 1: Register WorktreeCreate and WorktreeRemove hooks in plugin.json</name>
  <files>.claude-plugin/plugin.json</files>
  <action>
    Add `WorktreeCreate` and `WorktreeRemove` hook entries to the `hooks` object in `.claude-plugin/plugin.json`, alongside the existing `SessionStart` hook.

    Hook structure follows Claude Code plugin hook format:
    ```json
    "WorktreeCreate": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "node \"${CLAUDE_PLUGIN_ROOT}/bin/grd-tools.js\" worktree-hook-create \"$WORKTREE_PATH\" \"$WORKTREE_BRANCH\" 2>/dev/null || true",
            "timeout": 10
          }
        ]
      }
    ],
    "WorktreeRemove": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "node \"${CLAUDE_PLUGIN_ROOT}/bin/grd-tools.js\" worktree-hook-remove \"$WORKTREE_PATH\" \"$WORKTREE_BRANCH\" 2>/dev/null || true",
            "timeout": 10
          }
        ]
      }
    ]
    ```

    Key design decisions:
    - Use `2>/dev/null || true` to make hooks non-blocking (errors never interrupt Claude Code)
    - Use `$WORKTREE_PATH` and `$WORKTREE_BRANCH` environment variables that Claude Code passes to worktree hooks
    - Timeout of 10 seconds (longer than SessionStart's 5s because branch rename involves git operations)
    - Preserve the existing `SessionStart` hook exactly as-is
  </action>

    Outputs: SessionStart: true WorktreeCreate: true WorktreeRemove: true

<task type="auto">
  <name>Task 2: Implement hook handler functions and wire routing with tests</name>
  <files>lib/worktree.js, bin/grd-tools.js, tests/unit/worktree.test.js</files>
  <action>
    **In `lib/worktree.js`**, add two new exported functions:

    **`cmdWorktreeHookCreate(cwd, worktreePath, worktreeBranch, raw)`:**
    1. Check if GRD is active: `fs.existsSync(path.join(cwd, '.planning'))`. If not, output `{ skipped: true, reason: 'no .planning directory' }` and return (no-op).
    2. Load config via `loadConfig(cwd)`. If `branching_strategy` is `'none'`, output `{ skipped: true, reason: 'branching disabled' }` and return (no-op).
    3. Read the current milestone from `getMilestoneInfo(cwd)`.
    4. Compute what GRD's branch name would be using `worktreeBranch()` helper. Compare with the actual `worktreeBranch` argument.
    5. If the branch name differs and follows a recognizable GRD pattern (i.e., contains a phase number), optionally rename the branch:
       - Extract phase number from the worktree path (look for pattern `grd-worktree-*-{phase}` or just log without rename if unrecognizable)
       - Use `execGit(worktreePath, ['branch', '-m', actualBranch, grdBranch])` to rename
    6. Log the creation by outputting JSON: `{ hooked: true, worktree_path, branch: finalBranch, renamed: boolean, original_branch }`.

    **Important constraints on branch rename:**
    - Only rename if the branch name does NOT already follow GRD convention (starts with `grd/`)
    - If rename fails (git error), continue without error — log `{ hooked: true, rename_failed: true, reason }`
    - Never block or error — this is a best-effort hook

    **`cmdWorktreeHookRemove(cwd, worktreePath, worktreeBranch, raw)`:**
    1. Check if GRD is active (same `.planning` check). If not, output skip.
    2. Check if `branching_strategy` is `'none'`. If so, output skip.
    3. Output `{ hooked: true, worktree_path, branch: worktreeBranch, action: 'remove_logged' }`.
    4. This is intentionally minimal — Phase 46 will add state cleanup here if needed.

    **In `bin/grd-tools.js`**, add routing for the new commands in the `routeCommand` switch/case:
    ```javascript
    case 'worktree-hook-create':
      cmdWorktreeHookCreate(cwd, args[0], args[1], raw);
      break;
    case 'worktree-hook-remove':
      cmdWorktreeHookRemove(cwd, args[0], args[1], raw);
      break;
    ```

    Import the new functions from `lib/worktree.js` where the other worktree imports exist.

    **In `tests/unit/worktree.test.js`**, add a new describe block `'Worktree Hook Handlers'` with tests:
    - `cmdWorktreeHookCreate skips when no .planning directory`
    - `cmdWorktreeHookCreate skips when branching_strategy is none`
    - `cmdWorktreeHookCreate logs creation when GRD is active`
    - `cmdWorktreeHookCreate does not rename branch already following GRD convention`
    - `cmdWorktreeHookRemove skips when no .planning directory`
    - `cmdWorktreeHookRemove skips when branching_strategy is none`
    - `cmdWorktreeHookRemove logs removal when GRD is active`

    Follow existing test patterns in worktree.test.js — use temp directories, mock config, and capture output via process.exit mock.
