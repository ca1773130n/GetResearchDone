<objective>
Add `base_branch` support to the execute-phase workflow so that when branching is enabled, the workflow checks out the configured base branch (default "main") and pulls from remote before creating the phase/milestone branch.

Purpose: Currently `execute-phase` runs `git checkout -b $BRANCH_NAME` from whatever branch is active, which after a completed milestone may be a stale phase branch. This fix ensures phase branches always fork from the latest main.

Output: Updated `loadConfig` with `base_branch` default, `cmdInitExecutePhase` with `base_branch` field, updated command template with graceful checkout-and-pull logic, and tests.
</objective>

    This ensures `base_branch` is only present (non-null) when branching is actually enabled, matching the requirement "when branching_strategy is not 'none'".

    **3. commands/execute-phase.md — Rewrite the handle_branching step:**

    Replace the entire `<step name="handle_branching">` section with logic that:

    a. When `branching_strategy` is `"none"`: skip (no change from current behavior).

    b. When `branching_strategy` is `"phase"` or `"milestone"`:
       - First, check for uncommitted changes: `git status --porcelain`. If non-empty, warn "Uncommitted changes detected on current branch. Continuing on current branch." and skip the checkout/pull entirely but still create/switch to the phase branch from current position.
       - Check current branch: `git rev-parse --abbrev-ref HEAD`. If already on `base_branch`, skip checkout.
       - If not on `base_branch`: `git checkout $BASE_BRANCH`. If this fails, warn "Could not checkout base_branch, continuing on current branch" and proceed to branch creation without pull.
       - Pull: `git pull origin $BASE_BRANCH`. If this fails (offline, no remote, conflicts), warn "Pull failed (offline or conflict), continuing from local base_branch" and proceed.
       - Then create/switch to phase branch: `git checkout -b "$BRANCH_NAME" 2>/dev/null || git checkout "$BRANCH_NAME"`

    Include the `base_branch` field in the JSON fields parsed from init (in the `initialize` step).

    Update the `Parse JSON for:` line in the initialize step to include `base_branch`.
  </action>
