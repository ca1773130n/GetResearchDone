    Create `lib/cleanup.js` as a new module following the established pattern of other lib/ modules (JSDoc on exported functions, `module.exports` at bottom).

    **Function 1: `getCleanupConfig(cwd)`**

    Read `phase_cleanup` section from `.planning/config.json`.

    1. Construct path: `path.join(cwd, '.planning', 'config.json')`.
    2. Try to read and parse the file. On failure (missing or invalid JSON), return defaults.
    3. Extract `phase_cleanup` section from parsed config.
    4. Merge with defaults: `{ enabled: false, refactoring: false, doc_sync: false }`.
    5. Return the merged config object.

    ```js
    const defaults = { enabled: false, refactoring: false, doc_sync: false };
    // ... read file, parse, merge ...
    return { ...defaults, ...(parsed.phase_cleanup || {}) };
    ```

    **Function 2: `analyzeComplexity(cwd, files, options = {})`**

    Run ESLint complexity analysis on specified files using the ESLint Node.js API.

    1. If `files` is empty, return `[]`.
    2. Set `threshold` from `options.threshold` or default to `10`.
    3. Use ESLint's Node.js API (`@eslint/js` is already a project dependency):
       - Import `ESLint` from the `eslint` package (already installed as devDependency).
       - Create an ESLint instance with `overrideConfigFile: true` and override rules to set `complexity: ['warn', threshold]`.
       - Run `eslint.lintFiles(absolutePaths)` where absolutePaths are resolved relative to cwd.
       - Parse results: for each message with ruleId `'complexity'`, extract file, line, function name (from message text), and complexity value (parse from message text using regex like `/complexity of (\d+)/`).
    4. Filter to only existing files (skip non-existent gracefully).
    5. Return array of `{ file: relativePathFromCwd, line, functionName, complexity }`.

    **Important:** The ESLint API is async. Use `execSync` to run ESLint as a CLI subprocess instead for simplicity (keeping the module synchronous like all other lib/ modules):
    ```js
    const { execSync } = require('child_process');
    // Run: npx eslint --no-eslintrc --rule 'complexity: ["warn", threshold]' --format json <files>
    ```
    Parse the JSON output. Extract violations from the `messages` array where `ruleId === 'complexity'`.

    Wrap in try/catch -- if ESLint is not installed or fails, return `[]` with no crash.

    **Function 3: `analyzeDeadExports(cwd, files, options = {})`**

    Detect unused exports by scanning `module.exports` declarations and searching for corresponding `require()` imports across the codebase.

    1. If `files` is empty, return `[]`.
    2. For each file in `files`:
       a. Read the file content.
       b. Extract exported names by parsing `module.exports = { ... }` pattern and individual `exports.name = ...` patterns.
       c. For each exported name, search the codebase (all `.js` files under cwd, excluding `node_modules/`) for `require` statements that destructure or use that name.
       d. Optionally exclude patterns from `options.excludePatterns` (e.g., `['tests/']`) when counting consumers.
       e. If an export has zero consumers (other than the file itself), mark it as a dead export.
    3. Return array of `{ file: relativePath, exportName, line }`.

    **Implementation approach for export extraction:**
    - Regex for `module.exports = {` block: capture names between `{` and `}` (handle multiline).
    - Regex for `exports.funcName`: capture `funcName`.
    - Regex for `module.exports = { a, b, c }`: split on commas.

    **Implementation approach for import scanning:**
    - Use `fs.readdirSync` recursively (or glob) to find all `.js` files.
    - For each JS file, read content and check for `require('.*/filename')` patterns that reference the exporting file.
    - Then check if the exported name appears in a destructure pattern `const { name } = require(...)` or is accessed via `mod.name`.

    Keep it simple and pattern-based (regex, not AST) for v0.1.0. Accept that edge cases exist; the goal is to catch obvious dead exports, not achieve 100% accuracy.

    **Function 4: `analyzeFileSize(cwd, files, thresholds = {})`**

    Check files against line-count thresholds.

    1. If `files` is empty, return `[]`.
    2. Set `maxLines` from `thresholds.maxLines` or default to `500`.
    3. For each file:
       a. Resolve to absolute path under cwd.
       b. If file does not exist, skip.
       c. Read file, count lines (split by `\n`).
       d. If `lines > maxLines`, add to violations.
    4. Return array of `{ file: relativePath, lines, threshold: maxLines }`.

    **Function 5: `runQualityAnalysis(cwd, phaseNum)`**

    Orchestrate all checks and return a structured quality report.

    1. Call `getCleanupConfig(cwd)`.
    2. If `!config.enabled`, return `{ skipped: true, reason: 'phase_cleanup not enabled' }`.
    3. Find modified files for the phase:
       a. Read PLAN.md files from `.planning/phases/{phaseNum}-*/` to get `files_modified` from frontmatter.
       b. If no plan files found, scan `lib/*.js` as fallback.
    4. Run all three analyses:
       - `complexityResults = analyzeComplexity(cwd, jsFiles)`
       - `deadExportResults = analyzeDeadExports(cwd, jsFiles)`
       - `fileSizeResults = analyzeFileSize(cwd, jsFiles)`
    5. Build structured report:
       ```js
       {
         phase: phaseNum,
         timestamp: new Date().toISOString().split('T')[0],
         summary: {
           total_issues: complexity.length + deadExports.length + fileSize.length,
           complexity_violations: complexity.length,
           dead_exports: deadExports.length,
           oversized_files: fileSize.length,
         },
         details: {
           complexity: complexityResults,
           dead_exports: deadExportResults,
           file_size: fileSizeResults,
         },
       }
       ```
    6. Return the report.
