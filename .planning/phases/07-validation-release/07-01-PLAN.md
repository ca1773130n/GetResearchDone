  <files>lib/utils.js, bin/grd-tools.js</files>
  <action>
    1. Add new validation helper functions to `lib/utils.js`:

       **`validatePhaseArg(phase)`** тАФ Validates CLI phase number arguments:
       - Must be a non-empty string
       - Must match pattern: digits with optional decimal (e.g., "1", "01", "02.1", "3")
       - May include a kebab-case suffix (e.g., "01-security-foundation") for commands that accept full phase identifiers
       - Reject null/undefined with message: "Phase number is required"
       - Reject invalid format with message: "Invalid phase number: must be digits with optional decimal (e.g., 01, 02.1)"
       - Return the validated phase string

       **`validateFileArg(filePath, cwd)`** тАФ Validates CLI file path arguments:
       - Must be a non-empty string
       - Reject null/undefined with message: "File path is required"
       - Delegate to existing `validateFilePath(filePath, cwd)` for path traversal/escape checks
       - Return the validated file path

       **`validateSubcommand(sub, validSubs, parentCmd)`** тАФ Validates subcommand arguments:
       - Must be a non-empty string
       - If null/undefined/empty, throw with message: "Subcommand required for '{parentCmd}'. Available: {validSubs.join(', ')}"
       - If not in validSubs array, throw with message: "Unknown {parentCmd} subcommand: '{sub}'. Available: {validSubs.join(', ')}"
       - Return the validated subcommand

       **`validateRequiredArg(value, argName)`** тАФ Validates required positional arguments:
       - If null/undefined/empty string, throw with message: "{argName} is required"
       - Return the value

       Export all four new functions from `lib/utils.js`.

    2. Wire validation into `bin/grd-tools.js` router by adding validation calls at each entry point.
       Import the new validators alongside existing imports from utils.

       - `case 'init':` when wf is `'execute-phase'`, `'plan-phase'`, `'verify-work'`, `'phase-op'` тЖТ validate `args[2]`
       - `case 'phase-plan-index':` тЖТ validate `args[1]`
       - `case 'phase-detail':` тЖТ validate `args[1]`

       **File path validation** тАФ Add `validateFileArg()` call in these cases:
       - `case 'verify-summary':` тЖТ validate `args[1]`
       - `case 'frontmatter':` тЖТ validate `args[2]` (file)
       - `case 'verify':` when sub is `'plan-structure'`, `'phase-completeness'`, `'references'`, `'artifacts'`, `'key-links'` тЖТ validate `args[2]`
       - `case 'summary-extract':` тЖТ validate `args[1]`

       **Subcommand validation** тАФ Replace bare error() calls with `validateSubcommand()`:
       - `case 'state':` тЖТ validate sub against `['load', 'get', 'patch', 'update', 'advance-plan', 'record-metric', 'update-progress', 'add-decision', 'add-blocker', 'resolve-blocker', 'record-session']`
       - `case 'template':` тЖТ validate sub against `['select', 'fill']`
       - `case 'frontmatter':` тЖТ validate sub against `['get', 'set', 'merge', 'validate']`
       - `case 'verify':` тЖТ validate sub against `['plan-structure', 'phase-completeness', 'references', 'commits', 'artifacts', 'key-links']`
       - `case 'phases':` тЖТ validate sub against `['list']`
       - `case 'roadmap':` тЖТ validate sub against `['get-phase', 'analyze']`
       - `case 'phase':` тЖТ validate sub against `['next-decimal', 'add', 'insert', 'remove', 'complete']`
       - `case 'milestone':` тЖТ validate sub against `['complete']`
       - `case 'validate':` тЖТ validate sub against `['consistency']`

       - `case 'init':` тЖТ validate wf against the full list of init workflows
       - `case 'tracker':` тЖТ validate args[1] against tracker subcommands

       **Important implementation notes:**

       - The state command without a subcommand defaults to `cmdStateLoad` тАФ preserve this behavior (validate subcommand only if args[1] is provided but invalid)

       - For commands that accept the `--raw` flag, validation must happen AFTER raw flag extraction (already the case since raw is extracted early)
       - Keep the router structure compact тАФ validation calls should be 1-2 lines, not verbose blocks
