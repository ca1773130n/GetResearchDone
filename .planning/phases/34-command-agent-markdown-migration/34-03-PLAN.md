<objective>
Migrate all 16 agent markdown files from hardcoded `.planning/` subdirectory paths to context-injected variables. Agents receive their context from orchestrator commands — the orchestrator passes resolved path variables in the PATHS block of the spawn prompt, and agents use those variables instead of hardcoding paths.

Purpose: Complete REQ-58 by ensuring every agent markdown file references milestone-scoped paths through context variables rather than hardcoded strings. Agents are spawned by commands that call init and have access to `research_dir`, `phases_dir`, `codebase_dir`, etc. The agents should reference these as `${research_dir}`, `${phases_dir}`, etc. in their instructions.

**IMPORTANT CONVENTION: Agent placeholder syntax**

Agents receive path variables in two ways:
1. **Agents that call init themselves** (grd-executor, grd-planner): Use `${research_dir}` (bash variable syntax) since they parse init JSON and set bash variables.
2. **Agents spawned by orchestrators** (the other 14 agents): The orchestrator resolves paths via init and passes them in a `PATHS:` block in the Task() spawn prompt. The agent markdown uses `${research_dir}` as a reference to "the research_dir value provided in the spawn context." Since agent markdown is read by an LLM (not a bash interpreter), `${research_dir}` serves as a semantic placeholder that the LLM resolves from the PATHS block provided in its spawn prompt.

Plans 34-01 and 34-02 update orchestrator commands to include PATHS blocks in their spawn prompts. This plan updates agent markdown to reference those variables instead of hardcoded paths. The two changes are complementary.

Output: 16 updated agent markdown files with all operational paths using context-injected variables
</objective>
