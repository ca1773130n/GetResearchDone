    Create `tests/unit/cleanup-noninterference.test.js` that validates the cleanup system does NOT interfere when disabled. The existing `tests/unit/cleanup.test.js` tests functional behavior. This file focuses on NON-INTERFERENCE.

    Import from tests/helpers/setup.js: captureOutput (if needed for command output comparison).
    Import from tests/helpers/fixtures.js: createFixtureDir, cleanupFixtureDir.

    **Test Group 1: Config-Gated Early Exit (runQualityAnalysis)**
    1. `enabled: false` returns `{skipped: true, reason: 'phase_cleanup not enabled'}` — already tested in cleanup.test.js, but here verify the return object has EXACTLY these fields and no others (no summary, no details, no phase, no timestamp).
    2. `phase_cleanup` section entirely missing from config.json → same skipped result.
    3. `phase_cleanup: {}` (section present but empty, enabled defaults to false) → skipped.
    4. `enabled: false` with `doc_sync: true, refactoring: true` → still skipped (enabled=false overrides everything).
    5. Malformed config.json (invalid JSON) → skipped (graceful degradation, not crash).

    **Test Group 2: No Filesystem Side Effects When Disabled**
    1. Create a temp dir with .planning/config.json (enabled: false), lib/ files, CHANGELOG.md, README.md with broken links.
    2. Record all file mtimes in the directory tree before calling runQualityAnalysis.
    3. Call runQualityAnalysis.
    4. Record all file mtimes after.
    5. Assert: NO file mtimes changed (no writes occurred). The only file read should be config.json itself.
    6. Assert: No new files created in the directory tree.

    **Test Group 3: No Filesystem Side Effects for generateCleanupPlan When Disabled**
    1. Create temp dir with phase_cleanup.enabled = false.
    2. Create a .planning/phases/01-test/ directory with existing plan files.
    3. Call generateCleanupPlan with a quality report that has total_issues > threshold.
    4. Assert: returns null.
    5. Assert: No new files created in the phases directory.
    6. Assert: Existing plan files are unchanged.

    **Test Group 4: Output Equivalence**
    1. Create two identical fixture dirs (A and B).
    2. Dir A: config has NO phase_cleanup section at all.
    3. Dir B: config has phase_cleanup: { enabled: false, doc_sync: true, refactoring: true }.
    4. For both, call runQualityAnalysis with the same phase number.
    5. Assert: both return identical results (both skipped with same reason).
    6. This proves that having the config section with enabled=false produces the same effect as not having it at all.

    **Test Group 5: Performance (No I/O Overhead)**
    1. Create temp dir with enabled: false.
    2. Populate with many files in lib/ (create 20+ dummy .js files to make a full scan slow).
    3. Time runQualityAnalysis with enabled: false.
    4. Assert: completes in <10ms (generous bound; expected <1ms since it only reads config.json then returns).
    5. For comparison sanity: time runQualityAnalysis with enabled: true on the same dir — should be measurably slower (but don't assert a specific time, just that disabled is faster).

    **Test Group 6: getCleanupConfig Defaults**
    1. Verify getCleanupConfig returns `{enabled: false, refactoring: false, doc_sync: false, cleanup_threshold: 5}` as defaults.
    2. Verify that a config with `phase_cleanup: {enabled: true}` returns enabled=true but other fields default.
    3. Verify custom cleanup_threshold is respected.

    Use os.tmpdir() based temp directories. Clean up in afterEach/afterAll. Each test must be independent.
