  <name>Task 1: Write MCP server unit tests for protocol, listing, and execution</name>
  <files>tests/unit/mcp-server.test.js</files>
  <action>
    Create `tests/unit/mcp-server.test.js` with comprehensive tests for the MCP server module.

    **Test setup:**
    - Import `McpServer`, `buildToolDefinitions`, `COMMAND_DESCRIPTORS` from `../../lib/mcp-server`
    - Create a fixture directory with minimal `.planning/` structure (use pattern from existing tests: `fs.cpSync` of fixture dir to temp, clean up in afterAll)
    - Instantiate `McpServer` with `{ cwd: fixtureDir }` for each test group

    **Test groups (organize with describe blocks):**

    **1. buildToolDefinitions() (schema generation tests):**
    - Returns an array with 50+ tool definitions
    - Each tool has `name` (string), `description` (string), `inputSchema` (object)
    - inputSchema has `type: "object"` and `properties` object
    - Tools with required params have `required` array in inputSchema
    - Tool names follow `grd_{command}_{subcommand}` convention
    - No duplicate tool names
    - Spot-check specific tools:
      - `grd_state_load` has no required params
      - `grd_state_get` has optional `section` param of type string
      - `grd_phase_add` has required `description` param
      - `grd_verify_plan_structure` has required `file` param
      - `grd_init_execute_phase` has required `phase` param

    **2. COMMAND_DESCRIPTORS (registry completeness):**
    - Has entries covering all command groups from routeCommand
    - Verify presence of key command families: state, verify, frontmatter, phase, init, tracker

    **3. handleMessage — initialize (protocol handshake):**
    - Valid initialize returns `{ protocolVersion, capabilities: { tools: {} }, serverInfo: { name: "grd-mcp-server", version } }`
    - Response has correct JSON-RPC structure: `{ jsonrpc: "2.0", id, result }`
    - Initialize with different protocol versions still works (backward compat)

    **4. handleMessage — notifications/initialized:**
    - Returns null (notifications don't get responses)

    **5. handleMessage — tools/list:**
    - Returns `{ tools: [...] }` with all tool definitions
    - Each tool in response matches buildToolDefinitions output

    **6. handleMessage — tools/call (execution):**
    - Call `grd_state_load` → returns content array with JSON text
    - Call `grd_validate_consistency` → returns content array with result
    - Call `grd_detect_backend` → returns content array with backend info
    - Call `grd_generate_slug` with `{ text: "Hello World" }` → returns content with slug
    - Call `grd_current_timestamp` with `{ format: "date" }` → returns content with date string
    - Successful calls return `{ content: [{ type: "text", text: "..." }] }`
    - The `text` field contains valid JSON (parseable)

    **7. handleMessage — error paths:**
    - Unknown method → `{ code: -32601, message: "Method not found" }`
    - Unknown tool name in tools/call → error with code -32601 or -32602
    - Missing required params → error with code -32602 and data listing missing params
    - Missing `method` field → error with code -32600
    - Missing `id` field for request → still produces error response
    - tools/call with no `name` → error response
    - tools/call with no `arguments` when tool has required params → error with missing params

    **8. Edge cases:**
    - Null message → error response
    - Empty object → error response
    - Non-object message → error response

    Use the existing test patterns from the codebase:
    - `jest.spyOn(process, 'exit')` if needed for capture
    - Fixture isolation with temp directories
    - JSON.parse on results to verify structure

    IMPORTANT: Tests should use the fixture .planning/ directory from `tests/fixtures/` (copy to temp). The fixture has enough structure (config.json, ROADMAP.md, STATE.md, phases/) for state_load and validate_consistency to work.
