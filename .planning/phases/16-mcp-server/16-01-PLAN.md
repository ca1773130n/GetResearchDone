<task type="auto">
  <name>Task 1: Create MCP server protocol handler with auto-generated tool registry</name>
  <files>lib/mcp-server.js</files>
  <action>
    Create `lib/mcp-server.js` implementing the MCP protocol over stdio (JSON-RPC 2.0). This is the core module — zero external runtime dependencies (Node.js built-ins only).

    **Command Registry (auto-generated tool schemas):**

    Build a TOOL_REGISTRY array at module load time by defining each GRD CLI command as an MCP tool definition. The registry MUST be constructed programmatically from a declarative command descriptor table — NOT by hardcoding individual MCP tool JSON objects. The descriptor table maps each CLI command/subcommand to its parameters with types, required flags, and descriptions.

    Structure each descriptor as:
    ```js
    { name: 'grd_state_load', description: '...', params: [ { name: 'section', type: 'string', required: false, description: '...' } ] }
    ```

    Then a `buildToolDefinitions()` function transforms descriptors into MCP-format tool definitions with proper JSON Schema `inputSchema` objects.

    Cover ALL commands from `bin/grd-tools.js routeCommand`:
    - `state` (load, get, patch, update, advance-plan, record-metric, update-progress, add-decision, add-blocker, resolve-blocker, record-session)
    - `resolve-model`, `find-phase`, `commit`
    - `verify-summary`, `verify` (plan-structure, phase-completeness, references, commits, artifacts, key-links)
    - `template` (select, fill), `frontmatter` (get, set, merge, validate)
    - `generate-slug`, `current-timestamp`, `list-todos`, `verify-path-exists`
    - `config-ensure-section`, `config-set`, `history-digest`
    - `phases` (list), `roadmap` (get-phase, analyze), `phase` (next-decimal, add, insert, remove, complete)
    - `milestone` (complete), `validate` (consistency), `progress`, `todo` (complete)
    - `scaffold`, `init` (all 21 workflows)
    - `phase-plan-index`, `state-snapshot`, `summary-extract`
    - `tracker` (all 12 subcommands)
    - `dashboard`, `phase-detail`, `health`, `detect-backend`
    - `long-term-roadmap` (parse, validate, display, mode, generate, refine, promote, tier, history)
    - `quality-analysis`

    Naming convention: `grd_{command}_{subcommand}` (e.g., `grd_state_load`, `grd_verify_plan_structure`, `grd_phase_add`). Top-level commands without subcommands: `grd_{command}` (e.g., `grd_dashboard`, `grd_health`).

    **MCP Protocol Implementation:**

    Implement a `McpServer` class with these methods:

    1. `constructor(options)` — Initialize with `cwd` (defaults to process.cwd()), store tool registry
    2. `handleMessage(jsonRpcMessage)` — Parse and route JSON-RPC 2.0 messages:
       - `initialize` → return server capabilities (tools) and server info
       - `notifications/initialized` → acknowledge (no response needed for notifications)
       - `tools/list` → return tool definitions from registry
       - `tools/call` → dispatch to command functions (implemented here too)
       - Unknown method → JSON-RPC error -32601
    3. `_handleInitialize(id, params)` — Return `{ protocolVersion: "2024-11-05", capabilities: { tools: {} }, serverInfo: { name: "grd-mcp-server", version: "0.1.0" } }`
    4. `_handleToolsList(id)` — Return `{ tools: this.toolDefinitions }`
    5. `_handleToolsCall(id, params)` — Look up tool by name, validate required params, execute, return result
    6. `_executeTool(toolName, args)` — The dispatcher. Map tool names back to actual lib/ function calls. Capture stdout (since cmd* functions call process.stdout.write + process.exit). Return captured output as JSON content.

    **Tool Execution Strategy (critical):**

    The existing cmd* functions call `output(result, raw, rawValue)` which writes to stdout and calls `process.exit(0)`, or `error(msg)` which writes to stderr and calls `process.exit(1)`. For MCP tool execution, we need to capture this output WITHOUT actually exiting the process.

    Implement `_executeTool` with a capture pattern:
    - Temporarily replace `process.stdout.write`, `process.stderr.write`, and `process.exit`
    - Call the target cmd* function
    - Catch the exit sentinel (process.exit mock throws)
    - Restore originals
    - Return `{ stdout, stderr, exitCode }`

    This is the same pattern used by `tests/helpers/setup.js` (captureOutput/captureError) but adapted for the MCP server runtime.

    **Error Handling:**

    - Parse errors (malformed JSON) → `{ code: -32700, message: "Parse error" }`
    - Invalid request (missing method/id) → `{ code: -32600, message: "Invalid Request" }`
    - Method not found → `{ code: -32601, message: "Method not found" }`
    - Invalid params (missing required) → `{ code: -32602, message: "Invalid params", data: { missing: [...] } }`
    - Internal error (command throws) → `{ code: -32603, message: "Internal error", data: { error: "..." } }`

    **Exports:**
    - `McpServer` class
    - `buildToolDefinitions()` function (for testing)
    - `COMMAND_DESCRIPTORS` array (for testing/inspection)

    IMPORTANT: Do NOT import from `bin/grd-tools.js`. Import directly from `lib/*.js` modules. The MCP server bypasses the CLI router entirely and calls lib/ functions directly.

<task type="auto">
  <name>Task 2: Create MCP server entry point with stdio transport</name>
  <files>bin/grd-mcp-server.js</files>
  <action>
    Create `bin/grd-mcp-server.js` — the executable entry point that wires `lib/mcp-server.js` to stdin/stdout as an MCP stdio transport.

    **Implementation:**

    ```
    #!/usr/bin/env node
    ```

    1. Import `McpServer` from `../lib/mcp-server`
    2. Create server instance: `new McpServer({ cwd: process.cwd() })`
    3. Read JSON-RPC messages from stdin (newline-delimited JSON):
       - Buffer incoming data on `process.stdin`
       - Split on newlines
       - Parse each line as JSON
       - Call `server.handleMessage(parsed)` for each
       - Write response to stdout as JSON + newline (if response is not null — notifications have no response)
    4. Handle stdin close gracefully (exit 0)
    5. Handle parse errors: write JSON-RPC error response with code -32700

    **Stdio Transport Details:**
    - Input: one JSON-RPC message per line on stdin
    - Output: one JSON-RPC response per line on stdout
    - No stderr output during normal operation (stderr reserved for debug/logging if needed)
    - Server runs until stdin closes (EOF)

    **Edge cases:**
    - Empty lines → skip
    - Malformed JSON → write parse error response, continue (don't crash)
    - stdin ends mid-message → handle gracefully

    Make the file executable: it should have the shebang line and work with `node bin/grd-mcp-server.js`.

    Keep this file THIN — all logic lives in lib/mcp-server.js. This file is just the stdio transport wiring (similar to how bin/grd-tools.js is just a CLI router).
  </action>
