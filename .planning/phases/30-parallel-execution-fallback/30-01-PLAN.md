    **Module header:**
    ```javascript
    /**
     * GRD Parallel Execution — Multi-phase independence validation, context building, mode selection
     *
     * Provides tooling for validating that requested phases can execute in parallel
     * (no dependency edges between them), building per-phase execution context with
     * worktree paths, and selecting parallel vs sequential mode based on backend capabilities.
     *
     * Depends on: lib/deps.js (buildDependencyGraph), lib/utils.js, lib/backend.js, lib/roadmap.js
     */
    const fs = require('fs');
    const os = require('os');
    const path = require('path');
    const { output, error, loadConfig, findPhaseInternal, getMilestoneInfo } = require('./utils');
    const { detectBackend, getBackendCapabilities } = require('./backend');
    const { buildDependencyGraph } = require('./deps');
    const { analyzeRoadmap } = require('./roadmap');
    const { runPreflightGates } = require('./gates');
    ```

    **Function 1: validateIndependentPhases(graph, requestedPhases)**
    - Input: dependency graph `{ nodes, edges }` from buildDependencyGraph, array of phase number strings
    - Output: `{ valid: boolean, conflicts?: Array<{from, to}>, phases: string[] }`
    - Implementation:
      - If requestedPhases.length <= 1, return { valid: true, phases: requestedPhases }
      - Create a Set of requested phase IDs for O(1) lookup
      - Iterate over graph.edges: if both edge.from AND edge.to are in the requested set, record as conflict
      - If conflicts found: return { valid: false, conflicts, phases: requestedPhases }
      - Otherwise: return { valid: true, phases: requestedPhases }
    - Note: Only checks DIRECT edges (not transitive paths). This is correct because the command template already receives parallel_groups from deps analysis for ordering. The validation here prevents the user from requesting two phases where one directly depends on the other.

    **Function 2: buildParallelContext(cwd, phaseNumbers)**
    - Input: cwd (project directory), array of phase number strings
    - Output: structured JSON context for the execute-phase command template
    - Implementation:
      1. Load config via `loadConfig(cwd)`
      2. Detect backend via `detectBackend(cwd)`, get capabilities via `getBackendCapabilities(backend)`
      3. Get milestone info via `getMilestoneInfo(cwd)`
      4. Determine mode:
         - If `capabilities.teams === true` AND `config.use_teams !== false`: mode = 'parallel'
         - Otherwise: mode = 'sequential'
      5. Build fallback_note:
         - If mode === 'sequential': `'Parallel execution available on Claude Code backend with teams enabled'`
         - Otherwise: null
      6. For each phase number, build a phase context object:
         - Call `findPhaseInternal(cwd, phaseNum)` to get phase info
         - If phase not found, return `{ error: 'Phase {N} not found' }`
         - Include: phase_number, phase_name, phase_slug, phase_dir (from findPhaseInternal)
         - Include: worktree_path (computed: `path.join(fs.realpathSync(os.tmpdir()), 'grd-worktree-' + milestone.version + '-' + phaseNum)`)
         - Include: worktree_branch (computed from config.phase_branch_template, consistent with context.js pattern)
         - Include: plans, incomplete_plans, plan_count from phaseInfo
      7. Build status_tracker:
         - `{ phases: { [phaseNum]: { status: 'pending' } } }` for each phase
      8. Return complete context:
         ```javascript
         {
           mode,                      // 'parallel' | 'sequential'
           fallback_note,             // string | null
           backend,
           backend_capabilities: capabilities,
           use_teams: config.use_teams,
           team_timeout_minutes: config.team_timeout_minutes,
           max_concurrent_teammates: config.max_concurrent_teammates,
           milestone_version: milestone.version,
           milestone_name: milestone.name,
           phases: phaseContexts,     // Array of per-phase context objects
           phase_count: phaseNumbers.length,
           status_tracker,            // Per-phase status tracking
         }
         ```

    **Function 3: cmdInitExecuteParallel(cwd, phaseNumbers, includes, raw)**
    - CLI command function following the established `cmdInit*` pattern
    - Implementation:
      1. Validate all phase numbers are provided (at least 1)
      2. Run preflight gates via `runPreflightGates(cwd, 'execute-phase', { phase: phaseNumbers[0] })` — reuse execute-phase gates
      3. If gates fail, output gate errors and return
      4. Call `analyzeRoadmap(cwd)` to get all phases
      5. If roadmap error or empty, output error and return
      6. Verify all requested phases exist in roadmap phases (by number)
      7. If any phase not found, output error listing missing phases
      8. Build dependency graph: `buildDependencyGraph(roadmapPhases)`
      9. Call `validateIndependentPhases(graph, phaseNumbers)`
      10. If not valid, output error with conflicts and return
      11. Call `buildParallelContext(cwd, phaseNumbers)` to get full context
      12. If context has error, output it and return
      13. Add independence_validated: true to context
      14. Output the context JSON

    };
